# Phase 6.1: Landing Page Redesign - Research

**Researched:** 2026-01-31
**Domain:** Landing page UX/UI with parallax scrolling, bento grids, and conversion optimization
**Confidence:** HIGH

## Summary

This phase transforms a basic landing page into a conversion-optimized experience using parallax travel photography, bento grid layouts, warm minimalist design, and social proof. The research confirms that this approach aligns with 2026 landing page trends, with specific technical patterns well-established for Framer Motion parallax effects, CSS Grid bento layouts, and accessibility.

The standard stack leverages existing dependencies (Framer Motion 11.11.17, Tailwind CSS v4, Astro 5 with React islands) with the addition of Lenis smooth scroll library for enhanced parallax performance. Implementation patterns are mature and well-documented, with clear best practices for mobile performance, accessibility (prefers-reduced-motion), and Core Web Vitals optimization.

Key challenges identified: parallax effects can drain mobile batteries and harm performance if not carefully optimized, bento grids require thoughtful responsive design to maintain visual impact on mobile, and social proof counters need backend API integration for real-time display.

**Primary recommendation:** Use Framer Motion's useScroll + useTransform hooks for layered parallax with CSS transforms (not layout properties), implement bento grid with CSS Grid's grid-auto-flow: dense for flexible card sizing, enforce prefers-reduced-motion fallbacks, and optimize images through Astro's Sharp integration with WebP/AVIF formats.

### Pattern 1: Multi-Layer Parallax with Scroll Progress
**What:** Track scroll position with useScroll, transform into different Y offsets per layer
**When to use:** Hero section with dramatic depth effect (user's locked decision)

**Example:**
```typescript
// Source: https://blog.olivierlarose.com/tutorials/background-image-parallax
import { useScroll, useTransform, motion } from 'framer-motion';
import { useRef } from 'react';

export default function ParallaxHero() {
  const container = useRef(null);

  const { scrollYProgress } = useScroll({
    target: container,
    offset: ['start start', 'end start'] // Start when top hits viewport top, end when bottom hits viewport top
  });

  // Different speeds for each layer - creates depth
  const y1 = useTransform(scrollYProgress, [0, 1], ['0vh', '50vh']);   // Slowest (background)
  const y2 = useTransform(scrollYProgress, [0, 1], ['0vh', '100vh']);  // Medium (midground)
  const y3 = useTransform(scrollYProgress, [0, 1], ['0vh', '150vh']);  // Fastest (foreground)

  return (
    <div ref={container} className="relative h-screen overflow-hidden">
      {/* Layer 1: Background - slowest movement */}
      <motion.div style={{ y: y1 }} className="absolute inset-0">
        <img src="/images/hero/layer-3-background.jpg" alt="" className="w-full h-full object-cover" />
      </motion.div>

      {/* Layer 2: Midground - medium speed */}
      <motion.div style={{ y: y2 }} className="absolute inset-0">
        <img src="/images/hero/layer-2-midground.jpg" alt="" className="w-full h-full object-cover opacity-80" />
      </motion.div>

      {/* Layer 3: Foreground - fastest movement */}
      <motion.div style={{ y: y3 }} className="absolute inset-0">
        <img src="/images/hero/layer-1-foreground.jpg" alt="" className="w-full h-full object-cover opacity-60" />
      </motion.div>
    </div>
  );
}
```

### Pattern 4: Prefers-Reduced-Motion Accessibility Fallback
**What:** Detect user motion preference and disable parallax for accessibility
**When to use:** ALWAYS - WCAG compliance requirement for scroll animations

**Example:**
```typescript
// Source: https://motion.dev/docs/react-accessibility
import { useReducedMotion } from 'framer-motion';

export default function AccessibleParallax() {
  const shouldReduceMotion = useReducedMotion();

  const y = useTransform(
    scrollYProgress,
    [0, 1],
    shouldReduceMotion ? [0, 0] : ['0vh', '150vh'] // Disable parallax if user prefers reduced motion
  );

  return (
    <motion.div style={{ y }}>
      {/* Content */}
    </motion.div>
  );
}
```

### Pattern 5: Lenis Smooth Scroll Integration
**What:** Wrap app in smooth scroll provider for butter-smooth parallax experience
**When to use:** Root component to enhance all scroll-based animations

**Example:**
```typescript
// Source: https://github.com/darkroomengineering/lenis + React integration tutorials
import { useEffect } from 'react';
import Lenis from 'lenis';

export default function SmoothScrollProvider({ children }) {
  useEffect(() => {
    const lenis = new Lenis({
      duration: 1.2,        // Scroll duration
      easing: (t) => Math.min(1, 1.001 - Math.pow(2, -10 * t)), // Smooth easing
      smoothWheel: true,    // Enable smooth wheel scrolling
      wheelMultiplier: 1.0, // Scroll speed multiplier
    });

    function raf(time) {
      lenis.raf(time);
      requestAnimationFrame(raf);
    }

    requestAnimationFrame(raf);

    return () => lenis.destroy();
  }, []);

  return <>{children}</>;
}
```

### Anti-Patterns to Avoid
- **Using layout properties (top, left, height, width) for parallax:** Triggers layout recalculation, causes jank. Use CSS transforms instead.
- **Parallax without prefers-reduced-motion check:** Causes motion sickness for users with vestibular disorders. Always provide fallback.
- **Heavy images without optimization:** 5MB JPEGs destroy mobile performance. Use WebP/AVIF, lazy load, serve responsive sizes.
- **Grid with fixed heights for all cards:** Creates awkward whitespace. Use auto-rows with span utilities for organic layout.
- **Animating on scroll event listener directly:** Causes 40-60 unnecessary calculations per second. Use Framer Motion's useScroll or throttle/debounce.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Smooth scroll behavior | Custom requestAnimationFrame loop | Lenis library | Handles input device differences, touch momentum, browser quirks, accessibility |
| Scroll progress tracking | Manual scroll event listeners + throttling | Framer Motion useScroll | Optimized RAF usage, handles edge cases (mobile safari, overscroll), provides normalized 0-1 values |
| Image optimization pipeline | Manual WebP conversion + srcset generation | Astro's Image component + Sharp | Automatic format selection (AVIF/WebP/fallback), responsive sizes, caching, lazy loading |
| Bento grid masonry layout | Masonry.js or custom positioning | CSS Grid with auto-flow: dense | Zero JS, native browser optimization, simpler maintenance, better accessibility |
| Reduced motion detection | Manual media query + state management | Framer Motion useReducedMotion | SSR-safe, updates on preference change, consistent API |
| Waitlist counter API | Custom real-time WebSocket solution | Simple polling or static count with server-side update | Real-time not needed for waitlist numbers, over-engineering creates maintenance burden |

**Key insight:** Scroll animations have many edge cases (mobile Safari overscroll, touch momentum, reduced motion, SSR hydration). Established libraries handle these; custom solutions miss subtle bugs that take weeks to discover.

## Common Pitfalls

### Pitfall 1: Parallax Destroys Mobile Performance and Battery
**What goes wrong:** Smooth 60fps parallax on desktop becomes janky 15fps battery drain on mobile devices
**Why it happens:** Mobile browsers optimize for battery life by degrading animation performance. Parallax constantly recalculates transforms on scroll, which is expensive on mobile CPUs. Touch scrolling doesn't trigger scroll events as smoothly as wheel scrolling.
**How to avoid:**
- Use CSS transforms (not top/left/width/height) - runs on compositor thread
- Add `will-change: transform` to parallax elements so browser pre-optimizes
- Consider disabling parallax on mobile entirely: `const isMobile = window.innerWidth < 768; const yTransform = isMobile ? 0 : transformedValue;`
- Use Lenis smooth scroll library which handles mobile touch momentum better than native
- Lazy load parallax images with `loading="lazy"` attribute
**Warning signs:**
- Users report "laggy scrolling" on phones
- Chrome DevTools Performance tab shows long frames (>16ms) during scroll
- Battery drain reported by mobile users

### Pitfall 2: Cumulative Layout Shift (CLS) from Images Without Dimensions
**What goes wrong:** Images load and push content down, creating jarring layout shifts that hurt Core Web Vitals scores
**Why it happens:** Browser doesn't know image dimensions until it loads, so reserves 0 height initially. When image loads, layout recalculates and content shifts. Parallax effects that change during scroll can worsen this if not using transforms.
**How to avoid:**
- Always specify width and height attributes on images (even if styled with CSS later)
- Use Astro's `<Image>` component which automatically reserves space: `<Image src={hero} alt="" width={1920} height={1080} />`
- Reserve space for dynamic content with min-height or aspect-ratio CSS
- Use `transform` property for all parallax movement (not layout-triggering properties)
- Ensure animations complete within 500ms to avoid CLS impact
**Warning signs:**
- Lighthouse CLS score above 0.1
- Content "jumps" as you scroll down the page
- Images suddenly appear and push buttons/text down

### Pitfall 3: Bento Grid Breaks Responsive Design on Mobile
**What goes wrong:** Beautiful desktop bento grid becomes awkward stacked cards or overlapping mess on mobile
**Why it happens:** Multi-column grids with varied sizes don't translate well to single-column mobile layouts. Fixed pixel heights create awkward whitespace. `col-span-2` becomes meaningless when only 1 column exists.
**How to avoid:**
- Use responsive grid-cols: `grid-cols-1 md:grid-cols-2 lg:grid-cols-4`
- Reset all spans on mobile: `col-span-1 md:col-span-2` so cards stack naturally
- Use auto-rows with min-height: `auto-rows-[minmax(200px,auto)]` for flexible heights
- Test at 375px width (iPhone SE), 768px (iPad), 1024px (desktop)
- Consider different bento layouts per breakpoint, not just rescaling
**Warning signs:**
- Horizontal scroll on mobile
- Cards overlapping each other
- Huge whitespace gaps between cards
- Content cut off at narrow widths

### Pitfall 4: Parallax Without Reduced-Motion Fallback Causes Motion Sickness
**What goes wrong:** Users with vestibular disorders experience nausea, dizziness, and discomfort from parallax animations
**Why it happens:** Inner ear sensitivity to visual motion can trigger physical symptoms. 35% of adults over 40 have vestibular dysfunction. Users enable `prefers-reduced-motion` setting, but sites ignore it.
**How to avoid:**
- ALWAYS check `useReducedMotion()` hook before applying parallax transforms
- Provide static alternative: full image visible without movement
- Test with OS-level reduced motion enabled (macOS: System Preferences > Accessibility > Display > Reduce Motion)
- Consider middle ground: subtle fade/scale instead of parallax for reduced motion users
- Apply to ALL animations, not just parallax (button hovers, section reveals, etc.)
**Warning signs:**
- User complaints about dizziness or nausea
- High bounce rate from users with accessibility settings
- WCAG 2.1 Level A compliance failure (Success Criterion 2.3.3)

### Pitfall 5: Social Proof Counter Doesn't Update or Shows Stale Data
**What goes wrong:** "Join 237 others" counter shows same number for weeks, or counts don't match actual waitlist
**Why it happens:** Static site generation means HTML is built once and served to all users. Real-time counter requires either: (a) client-side API call on every page load (slow), (b) frequent rebuilds (expensive), or (c) edge function (complex). Developers forget to invalidate cache when waitlist grows.
**How to avoid:**
- Use static count updated during build: read from database, bake into HTML
- Set cache TTL for landing page (revalidate every 1 hour, rebuild if count changes significantly)
- Use client-side fetch ONLY for the counter, not whole page: `useEffect(() => fetch('/api/waitlist/count'))`
- Round to nearest 10 or 100 for larger numbers ("Join 2,400+ others") to avoid appearing stale
- Consider "Join Blake, Zack, and 2,368 others" pattern with recent names + count
**Warning signs:**
- Counter shows same number after multiple signups
- Counter jumps by large amounts unpredictably
- Slow initial page load due to client-side fetch blocking render

### Pitfall 6: Inter Font Loading Causes FOUT (Flash of Unstyled Text)
**What goes wrong:** Page loads with system font, then "jumps" to Inter font once it loads, creating jarring visual shift
**Why it happens:** Web fonts load asynchronously. Browser shows fallback font immediately to avoid blank text, then swaps to web font when ready. Inter is a large variable font family (multiple weights).
**How to avoid:**
- Use `font-display: swap` with size-adjust CSS to minimize layout shift
- Preload critical font weights: `<link rel="preload" href="/fonts/Inter-var.woff2" as="font" type="font/woff2" crossorigin>`
- Limit loaded weights to only what's used (Regular 400, Semibold 600, Bold 700)
- Use system font stack as fallback with similar metrics: `font-family: Inter, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`
- Consider self-hosting Inter instead of Google Fonts for faster loading
**Warning signs:**
- Text "jumps" or changes size when fonts load
- Lighthouse complains about render-blocking fonts
- Users on slow connections see system font for 1-2 seconds

### Social Proof with Waitlist Counter (Client-Side Fetch)
```tsx
// Pattern: Fetch count on client, static content server-rendered
import { useState, useEffect } from 'react';

export default function SocialProof() {
  const [count, setCount] = useState<number | null>(null);

  useEffect(() => {
    fetch('/api/waitlist/count')
      .then(res => res.json())
      .then(data => setCount(data.count))
      .catch(() => setCount(null)); // Graceful failure
  }, []);

  return (
    <div className="text-center py-8">
      <p className="text-gray-600 text-lg">
        {count !== null ? (
          <>Join <strong className="text-orange-500">{count.toLocaleString()}</strong> others on the waitlist</>
        ) : (
          <>Join the waitlist for early access</>
        )}
      </p>
    </div>
  );
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Separate Framer Motion package | Motion.dev unified package | 2024-2025 | Framer Motion migrated to motion.dev domain, documentation URLs changed, same API |
| JPEG images | WebP/AVIF with fallback | 2023-2026 | 25-50% smaller file sizes, Astro handles conversion automatically |
| CSS background-attachment: fixed | JavaScript scroll transforms | 2020-2026 | Fixed attachment has poor mobile support and jank, JS transforms are smoother with libraries |
| Masonry.js for grids | CSS Grid auto-flow: dense | 2022-2026 | Native CSS solution, zero JS, better performance and accessibility |
| Custom scroll listeners | Framer Motion useScroll hook | 2021-2026 | Better performance (RAF optimized), handles edge cases, cleaner API |
| Variable fonts considered "experimental" | Variable fonts standard practice | 2023-2026 | Inter variable font offers flexibility without multiple file downloads |
| Real-time WebSocket counters | Polling or static counts | Ongoing | Real-time adds complexity for minimal UX benefit on waitlist counters |

**Deprecated/outdated:**
- `background-attachment: fixed` for parallax: Poor mobile performance, use transform-based JS parallax instead
- Google Fonts hosted fonts: Self-hosting is faster (one fewer DNS lookup), better privacy
- Parallax without reduced-motion check: WCAG 2.1 compliance requirement since 2018, no longer optional

## Open Questions

Things that couldn't be fully resolved:

1. **Optimal parallax layer count for performance vs. impact**
   - What we know: 3-5 layers is common pattern, more layers = more calculations per scroll frame
   - What's unclear: Exact performance budget for target devices (mobile throttling scenarios)
   - Recommendation: Start with 3 layers (background, midground, foreground), measure performance with Chrome DevTools on throttled connection, add more only if smooth

2. **Best source for travel photography that shows "friends in beautiful places"**
   - What we know: Unsplash and Pexels offer free stock photos, both have "group travel" categories
   - What's unclear: Specific search terms to find photos showing BOTH people AND destinations (not one or the other)
